# Архитектура и ограничения BudgetBasket

## Назначение системы

BudgetBasket поддерживает процесс бюджетирования по модулям: сотрудник создаёт заявку и плановые строки, экономист проверяет их и фиксирует результаты, администратор ведёт пользователей, модули и справочники. Статусы, права и возможность редактирования определяются backend, а не интерфейсом.

## Состав приложения

```text
Браузер
  │ HTTP
  ▼
frontend (React + Vite, :5173)
  │ REST API
  ▼
backend (FastAPI, :8000)
  ├── PostgreSQL (:5432 внутри Compose, :5433 на хосте)
  ├── SeaweedFS через S3 API (:8333)
  └── file_guard (внутренняя сеть, :8080)
        └── ClamAV
```

`frontend` отвечает только за представление и вызовы API. Бизнес-правила, авторизация, работа с заявками и вложениями находятся в `backend`. Метаданные хранятся в PostgreSQL, бинарные данные вложений — в SeaweedFS. Клиент получает файлы только через авторизованный endpoint backend и не видит внутренние ключи S3.

## Модули репозитория

| Каталог | Ответственность |
| --- | --- |
| `frontend/` | React-интерфейс, TypeScript DTO, API-клиент, страницы и компоненты |
| `backend/app/routers/` | HTTP endpoints FastAPI |
| `backend/app/services/` | Бизнес-логика, права доступа, файлы, импорт и экспорт |
| `backend/app/repositories/` | Доступ к PostgreSQL |
| `backend/alembic/` | Миграции схемы БД |
| `backend/db/init.sql` | Справочная SQL-схема; поддерживается синхронно с миграциями |
| `file_guard/` | Изолированная проверка загрузок |
| `docker-compose.yml` | Локальный стек и сетевые границы сервисов |

## Поток загрузки файла

```text
Пользователь → backend (авторизация и права) → file_guard → backend → SeaweedFS + PostgreSQL
```

До положительного результата `file_guard` объект не загружается в SeaweedFS и его метаданные не сохраняются в PostgreSQL. Если проверка отклоняет файл, API отвечает `400`; если сервис проверки недоступен или не готов — `503`. Сообщение об ошибке включает имя файла.

## Сетевые границы Docker Compose

- Сеть `app`: frontend, backend и `file_guard`.
- Сеть `data`: backend, PostgreSQL и pgAdmin; она внутренняя.
- Сеть `storage`: backend и SeaweedFS; она внутренняя.
- `file_guard` не публикует порт на хост и не подключён к сетям PostgreSQL или SeaweedFS.

## Ограничения вложений

- Поддерживаются PDF, PNG, JPG/JPEG, XLSX и DOCX; импорт справочников принимает только XLSX.
- Максимальный размер одного файла — 25 МиБ.
- Пустые файлы и несовпадение расширения, MIME-типа и фактического содержимого отклоняются.
- PDF проверяются на повреждение, шифрование и активное содержимое.
- DOCX/XLSX проверяются как ZIP-контейнеры: блокируются макросы, path traversal, опасные внешние связи и аномальные архивы.
- PNG/JPEG полностью декодируются и ограничены по размерам изображения.
- При настройках по умолчанию `file_guard` требует доступный ClamAV; пока антивирус не готов, сервис проверки не готов к приёму файлов.

Параметры этих правил задаются переменными `FILE_GUARD_*` в `.env.example` и `docker-compose.yml`.

## Инженерные ограничения

- PostgreSQL — единственный runtime-источник реляционных данных; локальные JSON-файлы не используются как база данных.
- SeaweedFS используется только через S3-совместимый API; внутренние сервисные порты не используются в прикладном коде.
- Новые изменения схемы требуют согласованного обновления ORM-моделей, миграций Alembic и `backend/db/init.sql`.
- Пароли не хранятся в открытом виде.
- Новые переменные окружения должны быть добавлены в `.env.example`.
- В production необходимо заменить локальные учётные данные и ограничить доступ к опубликованным портам.
